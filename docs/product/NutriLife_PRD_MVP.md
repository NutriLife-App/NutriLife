# NutriLife — Product Requirements Document (PRD) for MVP

## 1. Загальна інформація

### 1.1 Назва продукту
NutriLife

### 1.2 Тип продукту
Mobile-first застосунок для персоналізованого харчування

### 1.3 Версія документа
v1.1

### 1.4 Статус
Робочий документ для проєктування та розробки MVP

### 1.5 Основна ідея
NutriLife допомагає користувачу формувати базовий персоналізований план харчування з урахуванням фізіологічних параметрів, харчової мети, дієтичних обмежень, бюджету та локальної доступності продуктів.

### 1.6 Контекст
Початкова концепція описувала продукт як PWA або вебзастосунок із мобільною адаптацією. Поточний MVP реалізується як mobile-first застосунок на Expo / React Native. Бізнес-логіка зберігається, а UX, навігація та структура модулів адаптуються під мобільний сценарій.

## 2. Проблема

Користувачі часто:
- не знають, як скласти раціон під свою мету;
- не розуміють, що купувати для щоденного або тижневого харчування;
- не можуть поєднати користь, бюджет і реалістичність;
- губляться між шаблонними порадами та перевантаженими fitness / wellness застосунками;
- не бачать простого зв’язку між своїми параметрами, планом харчування і прогресом.

NutriLife має прибрати цю складність і перетворити її на простий сценарій:
заповнив базові дані → отримав реалістичний план → побачив список покупок → відстежуєш прогрес.

## 3. Мета продукту

### 3.1 Продуктова мета
Створити MVP, який дозволяє користувачу пройти ключовий сценарій персоналізованого харчування без перевантаження функціями.

### 3.2 Бізнес-мета
Перевірити, чи є попит на nutrition-first продукт, у якому користувач:
- проходить онбординг;
- взаємодіє з планом харчування;
- користується списком покупок;
- повертається в застосунок;
- бачить цінність базового платного рівня.

### 3.3 Гіпотеза MVP
Якщо дати користувачу простий персоналізований сценарій харчування з урахуванням мети, обмежень і бюджету, то він сприйматиме NutriLife як практичний щоденний інструмент, а не як перевантажений застосунок про “правильне життя”.

## 4. Цільова аудиторія

### 4.1 Primary audience
- люди, які хочуть схуднути без складних систем;
- люди, які хочуть підтримувати вагу;
- люди, які хочуть покращити раціон у межах доступного бюджету.

### 4.2 Secondary audience
- користувачі з дієтичними обмеженнями;
- родини;
- користувачі, яким важлива локальна доступність продуктів;
- люди, які хочуть бачити зрозумілий прогрес без медичного або спортивного перевантаження.

### 4.3 Нецільова аудиторія для MVP
- професійні спортсмени з потребою в глибокій спортивній нутриціології;
- користувачі, яким потрібні медичні рекомендації;
- користувачі, які очікують автоматичних інтеграцій з wearable-пристроями;
- великі групи, команди або підрозділи.

## 5. Ціннісна пропозиція

NutriLife:
- фокусується саме на харчуванні;
- враховує бюджет і доступність продуктів;
- дає простий UX без зайвого когнітивного навантаження;
- будується як практичний щоденний інструмент;
- поєднує профіль, план, покупки і прогрес в одному потоці.

Ключова обіцянка:
допомогти користувачу швидко отримати персоналізовану основу для харчування, яку можна зрозуміти, використати і підтримувати в реальному житті.

## 6. Платформи та формат MVP

### 6.1 Поточний формат
MVP реалізується як mobile-first застосунок на:
- Expo
- React Native
- Expo Router
- TypeScript

### 6.2 Подальший розвиток
- web companion або landing;
- адміністративна панель;
- production build для Google Play;
- production build для App Store.

### 6.3 Причина mobile-first підходу
NutriLife є продуктом регулярного використання. Мобільний формат краще підтримує звичку, швидкий доступ, майбутні нагадування та локальне збереження даних.

## 7. Межі MVP

### 7.1 In scope
- welcome screen;
- вибір пакета використання;
- вибір тарифного рівня;
- багатокроковий онбординг;
- базовий профіль користувача;
- home dashboard;
- meal plan screen;
- recipes screen;
- grocery list screen;
- progress screen;
- settings screen;
- mock data;

### 7.2 Out of scope

- складна логіка замін страв;
- медичні рекомендації.

## 8. Основні сценарії використання

### 8.1 Базовий happy path
1. Користувач відкриває застосунок.
2. Бачить welcome screen.
3. Обирає пакет використання.
4. Обирає тариф.
5. Проходить онбординг.
6. Потрапляє на Home.
7. Переглядає meal plan.
8. Переглядає recipes.
9. Переглядає grocery list.
10. Позначає куплені продукти.
11. Додає запис ваги.
12. Повертається у застосунок.

### 8.2 Secondary scenarios
- користувач редагує дані в Settings;
- перемикає daily / weekly plan;
- натискає Replace meal і бачить керований placeholder;
- дивиться estimated grocery cost;
- перемикає month / year у Progress;
- змінює мову застосунку;
- користується dark mode.

## 10. Опис функціональних модулів

### 10.1 Welcome
Мета: швидко пояснити, що таке NutriLife, і перевести користувача до вибору пакета.

Контент:
- назва продукту;
- короткий підзаголовок;
- основний CTA Get started;
- secondary CTA Sign in як placeholder.

### 10.2 Package and Plan Selection
Мета: дати зрозумілий стартовий вибір режиму використання і показати базову тарифну логіку.

Package type:
- Individual
- Family

Plan tier:
- Free
- Basic
- Premium

Premium:
- показується в UI;
- позначається як Coming later;
- не реалізується функціонально в MVP.

### 10.3 Onboarding
Мета: зібрати мінімальний набір даних, достатній для персоналізованого mock experience.

Поля:
- Email
- Date of birth
- Gender
- Height
- Current weight
- Physical activity level
- Nutrition goal
- Dietary restrictions
- Geolocation placeholder
- Daily or weekly food budget

### 10.4 Home Dashboard
Мета: дати користувачу відчуття, що застосунок його “зрозумів” і вже може запропонувати персоналізований старт.

Вміст:
- greeting;
- package type;
- plan tier;
- current goal;
- summary card;
- quick actions.

### 10.5 Meal Plan
Мета: показати базовий персоналізований план харчування у зрозумілій формі.

Вміст:
- daily / weekly toggle;
- meal cards;
- nutrition summary;
- Replace meal button як UI-функція;
- mock data.

### 10.6 Recipes
Мета: дати користувачу список страв, пов’язаних із планом або доступних для перегляду.

Вміст:
- recipe list;
- recipe card;
- nutrition data;
- tags;
- ingredients;
- instructions placeholder;
- photo placeholder.

### 10.7 Grocery List
Мета: перетворити meal plan на практичний список покупок.

Вміст:
- items grouped by category;
- estimated price;
- cheapest offer placeholder;
- purchased toggle;
- total estimated cost.

### 10.8 Progress Tracking
Мета: дати користувачу простий інструмент для фіксації прогресу.

Вміст:
- manual weight entry;
- monthly / yearly toggle;
- simple chart placeholder;
- history list або summary.

### 10.9 Settings
Мета: дати користувачу місце для перегляду та базового редагування своїх параметрів.

Вміст:
- profile info;
- dietary preferences;
- budget;
- package info;
- logout placeholder;
- theme settings;
- language settings.

## 11. Theme requirements

### 11.1 Стратегія тем
NutriLife є light-first продуктом із повною підтримкою dark mode.

### 11.2 Що це означає
- світла тема є основною брендово-візуальною подачею;
- темна тема підтримується на рівні всіх основних екранів;
- інтерфейс має бути цілісним в обох режимах;
- MVP може працювати за принципом follow system;
- ручний перемикач System / Light / Dark може бути реалізований або підготовлений в Settings.

### 11.3 Причина вибору
Для nutrition / wellness продукту світла тема краще підтримує відчуття чистоти, читабельності та природності, а темна тема додає комфорт і платформну якість.

## 12. Localization requirements

### 12.1 Мови MVP
- Primary language: Ukrainian
- Secondary language: English

### 12.2 Правила
- українська є основною мовою інтерфейсу;
- англійська підтримується з першої версії архітектурно;
- user-facing strings не хардкодяться в компонентах;
- усі тексти проходять через localization layer;
- у майбутньому система має дозволяти легко додати нові мови.

### 12.3 Налаштування
- MVP може слідувати системній мові пристрою;
- у Settings має бути закладене місце для ручного вибору мови або явний placeholder.

## 15. Вимоги до дизайну

### 15.1 Візуальний напрям
- light-first wellness UI;
- dark mode support;
- сучасний чистий інтерфейс;
- м’які зелені акценти;
- теплі нейтральні відтінки;
- достатні відступи;
- зручні tap targets.

### 15.2 Чого уникати
- стилістики generic fitness app;
- надмірної технічності;
- строкатих кольорів;
- надто щільних форм;
- випадкових веб-компонентів у mobile UI.

### 15.3 UX принципи
- простий шлях вперед;
- зрозумілі CTA;
- мінімум тупиків;
- прогнозована навігація;
- інтерфейс має підтримувати довіру.

## 16. Нефункціональні вимоги

### 16.1 Технічні
- застосунок компілюється без unresolved imports;
- маршрути працюють стабільно;
- структура коду модульна;
- типізація TypeScript послідовна;
- theme system не ламає екрани;
- localization layer не розмазаний по компонентах.

### 16.2 Продуктові
- перший сценарій проходиться без допомоги розробника;
- mock-логіка виглядає зв’язною;
- користувач відчуває персоналізацію навіть у демо-версії.

### 16.3 UX
- форми зручні на мобільних;
- час до розуміння цінності короткий;
- користувач не губиться між екранами;
- застосунок лишається читабельним у light і dark.

## 17. Метрики успіху для MVP

### 17.1 Product validation metrics
- completion rate онбордингу;
- частка користувачів, які доходять до Home;
- частка користувачів, які відкривають Meal Plan;
- частка користувачів, які відкривають Grocery List;
- частка користувачів, які додають progress entry.

### 17.2 UX metrics
- кількість критичних помилок на ключовому шляху;
- кількість зламаних маршрутів;
- кількість місць, де користувач не розуміє, що робити далі;
- кількість theme / localization inconsistencies.

### 17.3 Business learning metrics
- чи зрозуміла різниця між Free і Basic;
- чи сприймається Meal Plan як ключова цінність;
- чи сприймається Grocery List як практична користь;
- чи є інтерес до Family package;
- чи є цінність англійської мови для демо та ширшої аудиторії.

## 18. Ризики

### 18.1 Продуктові ризики
- MVP стане занадто широким;
- продукт втратить фокус;
- користувач не зрозуміє різницю між пакетом і тарифом;
- персоналізація буде відчуватися поверхневою.

### 18.2 UX ризики
- онбординг буде задовгим;
- світла і темна тема будуть стилістично різними;
- двомовність буде впроваджена хаотично.

### 18.3 Технічні ризики
- надто раннє підключення бекенду;
- спроба одразу зробити web + mobile + backend + subscriptions;
- хардкод текстів у компонентах;
- дублювання темних / світлих стилів без системи.

## 19. Відкриті питання
- чи буде Free реально доступним у першому публічному тесті;
- чи Family package у MVP лише як UI або з окремими сценаріями;
- budget вводиться як daily, weekly або обидва варіанти;
- як визначатиметься локальна доступність продуктів у майбутньому;
- чи буде ручний вибір теми в першому білді;
- чи буде ручний вибір мови в першому білді;
- які екрани мають пріоритет для англійської на старті.

## 21. Definition of Done для MVP
MVP можна вважати готовим, якщо:
- є повний прохідний сценарій від Welcome до Progress;
- усі основні екрани створені й працюють;
- routing стабільний;
- onboarding дані зберігаються в поточному state;
- Home показує персоналізований summary;
- Meal Plan відображає логічний mock plan;
- Grocery List дозволяє відмічати покупки;
- Progress дозволяє додати запис ваги;
- Settings відображає дані профілю;
- світла тема виглядає цілісно;
- dark mode не ламає продукт;
- українська локалізація є основною;
- англійська може бути підключена без переробки архітектури;
- продукт читається саме як nutrition app.
